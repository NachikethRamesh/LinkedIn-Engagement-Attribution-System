# Foundation Notes: social-attribution-engine

## Why these tables exist

- `posts`: canonical record of published LinkedIn posts. This is the content origin point for downstream engagement.
- `social_events`: granular engagement actions (reaction/comment/share) tied to posts and timestamped for timeline analysis.
- `accounts`: normalized company-level entity used for account-based attribution and pipeline reporting.
- `contacts`: person-level records attached to accounts to represent known stakeholders and social actors.
- `website_events`: post-engagement web behavior, including both known account traffic and anonymous traffic for future identity stitching.
- `opportunities`: CRM-style pipeline objects that eventually need influence scoring.
- `account_intent_scores`: derived output table for future scoring jobs (left empty by base seeding).
- `opportunity_influence`: derived output table for future attribution jobs (left empty by base seeding).
- `imports_log`: operational table tracking ingestion runs, row counts, and warnings.
- `social_posts` / `social_comments` / `social_engagement_actors` / `social_engagement_events`:
  normalized actor-level social graph used for person-level engagement traceability while preserving legacy `posts`/`social_events` compatibility.
- `social_post_metrics_snapshots` / `social_comment_metrics_snapshots`:
  snapshot tables for aggregate metrics where APIs expose counts but not actor identities.
- `pipeline_runs`: orchestration run tracking table for job status, timing, params, and output metrics.
- `social_event_matches`: derived identity-resolution output linking social events to matched contacts/accounts.
- `account_intent_scores`: derived scoring output generated by the deterministic intent scoring engine.
- `opportunity_influence`: derived opportunity-level influence attribution output generated by the attribution engine.

## How this supports a later attribution engine

- The schema captures chronological data needed for attribution windows: social engagement events, web sessions, and opportunity creation.
- `social_events.post_id` and `posts.created_at` support post-level influence analysis.
- `social_events.actor_company_raw` and `contacts` provide matching inputs for account/contact resolution.
- `social_engagement_events` exposes object-level engagement context (`post/comment/reply`) and actor-resolution status (`resolved/aggregate_only`) for deeper attribution diagnostics and future multi-platform expansion.
- `website_events` links campaign context (`utm_source`, `utm_campaign`) with account activity.
- `account_intent_scores` and `opportunity_influence` are storage layers for model output so attribution logic can evolve without rewriting source tables.
- Base seeding intentionally excludes both derived tables to avoid circular demo logic.
- `social_event_matches` stores deterministic entity matching output so downstream scoring can use audited identity signals.

## Source vs derived tables

- Source/input tables seeded in this foundation:
  `posts`, `social_events`, `accounts`, `contacts`, `website_events`, `opportunities`.
- Derived/output tables intentionally left empty after seeding:
  `account_intent_scores`, `opportunity_influence`, `social_event_matches`.
  They are populated by downstream jobs (`run_matching`, `run_scoring`, `run_attribution`) rather than seed scripts.

## Assumptions in this MVP foundation

- LinkedIn engagement is represented with three event types: `reaction`, `comment`, `share`.
- `actor_company_raw` is intentionally denormalized because company extraction quality will vary in early ingest stages.
- `website_events.account_id` can be null to represent anonymous traffic before identity resolution.
- Opportunity stage names are simple CRM-like strings, not a strict enum yet.
- Scoring and influence outputs are produced later by dedicated jobs, not by the foundational seed script.

## Intentionally deferred to later phases

- Attribution logic and scoring algorithms.
- Jobs that populate `account_intent_scores` and `opportunity_influence`.
- Identity resolution job that populates `social_event_matches`.
- Identity resolution and enrichment workflows.
- API layer, job scheduler, and event ingestion connectors.
- Authentication/authorization and multi-tenant controls.
- Cloud deployment, observability, and production hardening.
